# План и замечания по Telegram-боту для удаления `deactivated_account` (обновлённый)

## Короткий вывод
Да, это реализуемо. Оптимальная схема под ваш запрос:
- бот постоянно работает на **Render Free**;
- при приближении к лимитам Render автоматически включает **Hybrid-mode** и переносит тяжёлые/фоновые задачи в **GitHub Actions**;
- данные о premium/сроках/настройках хранятся в **бесплатной внешней БД без sleep-режима**;
- /premium берёт ссылки из `GitHub Secrets` (`TARIF_MESSAGE_1..N`) и подставляет их в inline-кнопки.

---

## 1) Ограничения и как их обойти

### 1.1 Telegram Bot API и лимит "50 000"
- В Bot API нет метода "получить сразу всех участников".
- Поэтому реальный максимум достигается так:
  1. накапливаем собственный реестр участников (`chat_member`, `my_chat_member`, сообщения, join-события);
  2. сканируем этот реестр батчами;
  3. для premium повышаем потолок до 50 000 **известных** пользователей на цикл.

> Важно: "50 000 всех участников чата с нуля" в чистом Bot API недоступно; доступно "до 50 000 известных системе участников".

### 1.2 Render Free (что учитывать)
Для Free-плана нужно закладывать типичные ограничения бесплатного хостинга:
- возможный sleep/простой при неактивности;
- ограниченные compute-ресурсы и квоты;
- риски задержек при cold start.

Из-за этого для стабильности лучше не держать тяжёлый full-scan внутри одного runtime-процесса.

### 1.3 Почему нужен Hybrid-mode
Hybrid-mode снимает нагрузку с Render:
- realtime-часть (команды, кнопки, быстрые ответы) остаётся на Render;
- тяжёлые фоновые задачи запускаются в GitHub Actions по расписанию;
- результат фоновой задачи бот забирает из БД и публикует отчёт.

## 2) Рекомендуемая архитектура (бесплатно + надёжно)

## 2.1 Компоненты
1. **Render Free Web Service**
   - webhook-приём обновлений Telegram;
   - обработка `/start`, `/premium`, `/settings`, `/dev`;
   - управление inline-кнопками и правами.

2. **GitHub Actions (scheduler/worker)**
   - cron-задачи сканирования;
   - фоновая обработка больших очередей (проверка участников, пересчёт remaining time, cleanup задач).

3. **Cloudflare D1 (SQLite-as-a-service) — основной storage**
   - полностью бесплатный слой хранения с постоянной записью/обновлением;
   - таблицы: `subscriptions`, `chat_settings`, `members_cache`, `scan_jobs`, `scan_reports`, `audit_log`, `roles`.

> Почему D1: бесплатный, стабильно доступный, без зависимости от "git-файлового" storage, лучше подходит под постоянные обновления, чем хранение JSON в репозитории.

## 3) Hybrid-mode: правила автоматического переключения

## 3.1 Триггеры включения Hybrid-mode
Если выполняется хотя бы одно условие, тяжёлые задачи выносятся в Actions:
- очередь фоновых задач > заданного порога (например, 1 000);
- длительность последнего скана > 20–30 секунд;
- CPU/RAM процессу близки к заданному soft-limit;
- чат имеет размер скана > 2 000 (free) или > 10 000 (premium) в одной итерации.

## 3.2 Что делает Render в Hybrid-mode
- мгновенно принимает команду пользователя;
- создаёт `scan_job` в БД;
- отвечает "задача поставлена в очередь";
- публикует отчёт по завершении воркера.

## 3.3 Что делает GitHub Actions в Hybrid-mode
- каждые 5–10 минут забирает `pending` jobs;
- обрабатывает батчами (например, по 500–1 000 пользователей);
- пишет прогресс и итоговый отчёт в БД;
- отмечает `done/failed`.

## 4) Команды и UX

## 4.1 `/start`
- приветствие;
- кнопка "➕ Добавить бота в чат" (`startgroup` link);
- проверка прав бота;
- статус: `Ready / Limited / Hybrid`.

## 4.2 `/premium`
- красиво оформленный текст (MarkdownV2/HTML):
  - 1 месяц — 199 ₽;
  - 3 месяца — 499 ₽ ~~599 ₽~~;
  - 6 месяцев — 959 ₽ ~~1194 ₽~~;
  - 12 месяцев — 1999 ₽ ~~2388 ₽~~.
- кнопки "Купить": ссылки берутся из secrets:
  - `TARIF_MESSAGE_1`
  - `TARIF_MESSAGE_3`
  - `TARIF_MESSAGE_6`
  - `TARIF_MESSAGE_12`
- дополнительная строка поддержки:
  - «Если не получается оформить подписку — обратитесь в [поддержку бота](https://t.me/kiojomi)».

## 4.3 `/settings`
- доступ:
  - владелец/админ — read/write;
  - помощник — read-only.
- пункты:
  - интервал проверки:
    - часы (free);
    - минуты/секунды (🔒 premium);
  - автоудаление frozen accounts (🔒 premium);
  - роли и помощники (добавление по ID);
  - просмотр текущих параметров.

## 4.4 `/dev` (скрытая)
- только для `DEV_TELEGRAM_IDS`;
- выдача premium по Telegram ID;
- выбор срока кнопками;
- запись в `subscriptions` + аудит в `audit_log`.

---

## 5) Логика сканирования и удаления

1. Планировщик выбирает чат и лимит:
   - free: до 2 000 на цикл;
   - premium: до 50 000 известных участников (батчево).
2. Проверка аккаунтов:
   - при обнаружении целевого аккаунта — `banChatMember`/ограничение по политике;
   - запись события в `scan_reports`.
3. Отчёт:
   - бот отправляет summary в чат;
   - удаляет своё отчётное сообщение через 2 минуты (`deleteMessage`).

---

## 6) Хранение данных (выбранный вариант)

## 6.1 Что храним
- `subscriptions`: user_id, plan, started_at, expires_at, remaining_seconds;
- `chat_settings`: интервалы, флаги premium-функций, режимы;
- `roles`: owner/admin/helper;
- `members_cache`: известные участники для сканирования;
- `audit_log`: операции `/dev`, изменения настроек.

## 6.2 Почему не хранить это в GitHub-JSON
- сложнее с конкурентной записью;
- медленнее для частых обновлений;
- неудобнее для атомарных операций по времени подписки.

## 8) GitHub Actions workflow (`.github/workflows/bot-hybrid.yml`)

Триггеры:
- `schedule` (каждые 5–10 минут);
- `workflow_dispatch`.

Шаги:
1. checkout;
2. setup runtime;
3. install dependencies;
4. `python worker_scan.py --mode=hybrid`;
5. запись статуса/логов.

Опционально:
- отдельный workflow `subscription-recalc.yml` (пересчёт remaining time и истечений подписок).

---

## 9) Практические замечания
1. Даже в premium-режиме реальный потолок зависит от того, насколько полно собран `members_cache`.
2. Для больших чатов обязательно батчинг + retry/backoff.
3. Нужен антидребезг, чтобы один и тот же пользователь не удалялся повторно в параллельных задачах.
4. `/dev` должен вести полный аудит (кто, кому, на какой срок выдал premium).
